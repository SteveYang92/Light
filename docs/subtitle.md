# 字幕制作指南

> 本文档按读者需求分层：
> - **第一部分**（字幕基础）：所有场景通用
> - **第二部分**（翻译）：输出语言 ≠ 源语言时阅读
> - **第三部分**（工作流）：按输出场景跳转对应章节
> - **第四部分**（QC & 导出）：所有人阅读

---

# 第一部分：字幕制作基础

---

## 一、字幕制作规范

### 1.1 字幕格式选择

| 格式 | 用途 | 建议 |
|---|---|---|
| `SRT` | 最通用 | 必须输出 |
| `WebVTT` | 网页播放器、HTML5 | 必须输出 |
| `ASS/SSA` | 双语、样式、位置控制 | 推荐输出 |
| `TTML/IMSC` | 广电、流媒体交付 | 按平台要求输出 |

### 1.2 字幕长度与行数

| 项目 | 中文 | 英文 |
|---|---:|---:|
| 最大行数 | 2 行 | 2 行 |
| 每行长度 | 建议不超过 16 个汉字 | 建议不超过 42 个字符 |
| 单条最短时长 | 不低于约 0.8 秒 | 不低于约 0.8 秒 |
| 单条最长时长 | 不超过 7 秒 | 不超过 7 秒 |
| 成人节目阅读速度 | 中文不超过约 9 字/秒 | 英文按平台规范控制 |
| 儿童节目阅读速度 | 中文不超过约 7 字/秒 | 更保守 |

### 1.3 时间轴同步规则

1. **入点**：尽量贴近说话开始，可允许 1 到 2 帧误差。
2. **出点**：不要机械地在声音结束瞬间消失；如果后面没有紧接字幕，可适当延后约半秒。
3. **字幕间隔**：避免极短闪烁间隔；相邻字幕要么保留足够间隔，要么做 chaining。
4. **镜头切点**：字幕尽量不要无必要跨镜头；台词没有跨镜头时，字幕也尽量在镜头切换前结束。
5. **重叠说话**：两人同时讲话时字幕最多两行，每行一个说话人；重要信息优先，不重要的插话可省略或压缩。
6. **回看校验**：必须完整播放回看，不能只看时间码。

## 二、字幕切分规则引擎

字幕不是按 ASR segment 原样切，而要重新切分。

### 2.1 切分原则

优先级：

1. 不切断完整语义
2. 不切断固定搭配
3. 不切断姓名、品牌名、数字单位
4. 避免一条字幕太长
5. 避免一条字幕显示太短
6. 避免跨镜头切点
7. 与说话节奏一致

### 2.2 中文切分规则

- 每条字幕 1 到 2 行
- 每行尽量不超过 16 个汉字
- 避免把短词孤立成一行
- 断句优先位置：逗号、顿号、句号、语义短语边界、连词前后

不推荐：

```text
我们今天主要讨论这个项目
的交付时间
```

推荐：

```text
我们今天主要讨论
这个项目的交付时间
```

### 2.3 英文切分规则

- 每行不超过 42 个字符
- 不拆：article + noun、adjective + noun、first name + last name、subject + verb、auxiliary + verb、phrasal verb
- 优先在标点后、连词前、介词前换行

不推荐：

```text
I want to talk about the
new product launch.
```

推荐：

```text
I want to talk about
the new product launch.
```

## 三、同步精度与时间轴修正

### 3.1 为什么需要强制对齐

ASR 的段落级时间戳通常不够细。要做到字幕和声音同步，最好使用**逐词时间戳**或 forced alignment。

### 3.2 同步算法

建议每条字幕的时间由其包含的词决定：

```
start = first_word.start
end = last_word.end + reading_padding
```

其中：
- `reading_padding` 可设为 `0.2` 到 `0.5` 秒
- 如果下一条字幕很近，则压缩 padding
- 如果遇到镜头切点，则出点贴近镜头切点
- 如果字幕阅读速度超标，则优先压缩文字，而不是无限延长时间

### 3.3 时间轴修正规则

```
if subtitle_duration < min_duration:
    extend end time if possible

if subtitle_duration > max_duration:
    split subtitle

if gap_to_next < min_gap:
    chain or merge

if crosses_shot_change and dialogue_does_not_cross:
    adjust to shot boundary

if reading_speed > limit:
    condense text or split
```

# 第二部分：翻译

> 只有当输出语言与源语言不同时，才需要阅读本部分。
> 翻译的核心问题是：**什么时候译？以什么为单位译？规则什么时候应用？**

---

## 四、字幕翻译策略

### 4.1 核心结论

翻译应放在**"源语言初步分段之后、目标语言最终字幕规则之前"**。

字幕规则分两类：

1. **时间轴 / 语义分段规则**：什么时候开始/结束、一条字幕包含哪句话、是否按停顿/说话人变化/镜头切换断开
2. **语言排版规则**：每行多少字、最多几行、阅读速度 CPS、是否需要压缩翻译、如何换行

翻译应放在二者之间。

### 4.2 为什么先分段再翻译？

不建议流程：

```
完整中文转录稿 → 整篇翻译成英文 → 再切字幕
```

原因：
- 翻译后很难准确对应原声时间轴
- 中英文语序不同，句子长度不同
- 容易出现英文字幕和中文语音不同步
- 说话人切换和停顿信息容易丢失

例如中文说：

```
其实我们今天主要想聊的是，人工智能在视频制作流程里面到底能帮我们解决哪些问题。
```

如果先整段翻译再切，可能得到：

```
Actually, what we mainly want to talk about today is what problems
artificial intelligence can help us solve in the video production process.
```

这句话太长，不适合字幕。后面再切会很困难。

更好的方式是先按声音和语义切成字幕单元：

```
其实我们今天主要想聊的是，
人工智能在视频制作流程里面
到底能帮我们解决哪些问题。
```

然后翻译成字幕化英文：

```
Today, we'll focus on how AI
can improve video production.
```

### 4.3 为什么不能锁死最终格式再翻译？

如果中文已经按中文字幕规则做成最终版，再逐条翻译，英文会因语序不同变成碎片：

```
1
What we mainly want to talk about today is

2
in the video production process

3
what problems AI can help us solve.
```

所以翻译的输入不应是单条显示 cue，而应是**更大的语义块 + 上下文信息**：
- 当前文本
- 前后句
- 时间长度
- 长度限制
- 是否要压缩
- 术语表
- 说话人信息

### 4.4 规则的分阶段应用

| 阶段 | 应用什么规则 | 说明 |
|---|---|---|
| 翻译前 | 初步规则 | 根据语音、停顿、说话人切基本语义单元，生成源语言草案（软确定） |
| 翻译时 | 约束规则 | 告知模型每条的时间长度和长度限制 |
| 翻译后 | 最终规则 | 按目标语言 CPS、行长、语义重新换行和压缩 |
| 导出前 | QC 规则 | 检查重叠、太短、太长、不同步、超行长 |

#### 可以较早锁定的规则

与声音和画面相关的规则应尽早确定。

| 类型 | 是否可早锁 | 说明 |
|---|---:|---|
| 语音开始时间 | 是 | 来自词级时间戳 |
| 语音结束时间 | 是 | 来自词级时间戳 |
| 说话人切换 | 是 | 不能跨说话人太多 |
| 明显长停顿 | 是 | 停顿通常是自然断点 |
| 镜头硬切 | 半锁 | 尽量不跨切点 |
| 字幕重叠限制 | 是 | 必须避免 |

#### 不应过早锁死的规则

与语言显示相关的规则翻译后可能要调整。

| 类型 | 是否可早锁 | 说明 |
|---|---:|---|
| 中文每条字幕具体切在哪里 | 不建议完全锁死 | 英文语序可能需要调整 |
| 中文换行位置 | 不应锁死 | 只是中文显示规则 |
| 中英文 cue 一一对应 | 不应锁死 | 会导致英文别扭 |
| 英文断行 | 翻译后决定 | 目标语言规则优先 |
| 英文压缩程度 | 翻译后决定 | 根据 CPS 和时长决定 |

### 4.5 核心原则：不要把"源语言显示行"当成"翻译单位"

错误方式：

```
中文显示 cue 1 → en cue 1
中文显示 cue 2 → en cue 2
中文显示 cue 3 → en cue 3
```

正确方式：把多条显示 cue 视为一个**翻译语义块**，翻译后再根据目标语言规则切行。

## 五、中英字幕生成流程

### 5.1 翻译通用流程

```
音频 → 原文转录 → 原文校对 → 翻译（按第四章原则）→ 字幕化压缩 → 时间轴对齐
```

### 5.2 中文源音频

```
中文 ASR → 中文校对 → 英文翻译 → 英文字幕压缩 → 中英时间轴共用或微调
```

建议：
- 中文字幕更贴近口语原文
- 英文字幕不逐字硬翻，要符合英语观看习惯
- 专有名词使用术语表
- 对中文长句进行语义压缩

### 5.3 英文源音频

```
英文 ASR → 英文校对 → 中文字幕翻译 → 中文字幕压缩 → 中英时间轴共用或微调
```

建议：
- 英文字幕可以更接近原话
- 中文字幕应更凝练
- 英文中的重复、语气词、填充词不必全部翻译

原文：
`Well, I mean, I think we should probably start with the budget first.`

中文字幕可处理为：
`我觉得应该先从预算开始。`

而不是：
`嗯，我是说，我认为我们可能应该先从预算开始。`

## 六、三级数据结构设计

建议系统内部不要只有一种 `subtitle segment`，而要区分三层：

```
词级单位 word → 语义单位 semantic unit → 显示字幕单位 subtitle cue
```

### 6.1 词级单位

用于同步。

```json
{
  "text": "人工智能",
  "start": 3.2,
  "end": 3.8,
  "speaker": "SPEAKER_00"
}
```

### 6.2 语义单位

用于翻译。

```json
{
  "unit_id": "u001",
  "start": 1.2,
  "end": 6.8,
  "speaker": "主持人",
  "zh_text": "我们今天主要想聊的是，人工智能在视频制作流程里面到底能帮我们解决哪些问题。",
  "word_ids": ["w001", "w002", "w003"],
  "source_cue_ids": ["zh_001", "zh_002", "zh_003"]
}
```

### 6.3 显示字幕单位

用于最终导出。英文和中文不必一一对应，而是与 `unit_id` 对应：

```json
{
  "cue_id": "zh_001",
  "unit_id": "u001",
  "start": 1.2,
  "end": 3.0,
  "zh": "我们今天主要想聊的是"
}
```

```json
{
  "cue_id": "en_001",
  "unit_id": "u001",
  "start": 1.2,
  "end": 4.2,
  "en": "Today, we'll focus on how AI"
}
```

这样中文分了 3 条、英文只需要 2 条，但它们仍属同一个语义单位，时间来自原声词级时间戳。

### 6.4 中间格式（JSON）

建议内部统一使用 JSON，不要直接以 SRT 作为主数据。

```json
{
  "media": {
    "filename": "input.mp4",
    "duration": 3620.52,
    "fps": 25
  },
  "speakers": [
    { "id": "SPEAKER_00", "name": "主持人", "confidence": 0.98 },
    { "id": "SPEAKER_01", "name": "嘉宾", "confidence": 0.91 }
  ],
  "cues": [
    {
      "id": 1,
      "start": 10.240,
      "end": 13.600,
      "speaker": "SPEAKER_00",
      "source_text": "欢迎大家来到今天的节目。",
      "zh_text": "欢迎大家来到今天的节目。",
      "en_text": "Welcome to today's program.",
      "words": [
        { "text": "欢迎", "start": 10.240, "end": 10.620 }
      ],
      "qc": {
        "reading_speed_ok": true,
        "line_length_ok": true,
        "sync_ok": true
      }
    }
  ]
}
```

# 第三部分：输出工作流

> 按你的输出需求选择对应场景。翻译步骤标注了「按第二部分执行」。

---

## 场景 A：源语言 → 同语言字幕输出

**适用**：只需要中文字幕或只需要英文字幕，不涉及翻译。

```
ASR 转录
  ↓
文本清洗、标点恢复
  ↓
按切分规则生成字幕 cue（按第一、二章）
  ↓
词级时间戳对齐
  ↓
时间轴修正（按第三章）
  ↓
自动 QC（按 7.3 硬性规则）
  ↓
人工审校（按 7.5 审校要点）
  ↓
导出 SRT / VTT
```

---

## 场景 B：源语言 → 目标语言单语字幕输出

**适用**：如中文音频 → 英文字幕，最终只输出一种语言。

关键要求：**目标语言字幕必须严格对齐源语言词级时间戳**，因为字幕时间由源语言语音决定。

```
ASR 转录
  ↓
词级时间戳对齐
  ↓
文本清洗、标点恢复
  ↓
建立语义 unit（按第六章）
  ↓
翻译（按第二部分第四、五章）
  ↓
目标语言字幕化压缩/切分（按第二章目标语言规则）
  ↓
时间轴对齐（基于源语言词级时间戳，按第三章）
  ↓
自动 QC（按 7.3 硬性规则，含 7.3.3 场景 B 附加规则）
  ↓
人工审校（按 7.5 审校要点，含翻译审校）
  ↓
导出 SRT
```

---

## 场景 C：双语字幕输出

**适用**：同时输出中英双语字幕，如同一 cue 双语或中英分轨。

```
ASR 转录
  ↓
词级时间戳对齐
  ↓
文本清洗、标点恢复
  ↓
建立语义 unit（按第六章）
  ↓
翻译（按第二部分第四、五章）
  ↓
目标语言字幕化压缩
  ↓
双语协调：
    - 合并过碎源语言 cue
    - 拆分过长目标语言 cue
    - 压缩目标语言
    - 必要时微调时间轴
  ↓
双语排版（同 cue 双语或中英分轨）
  ↓
自动 QC（按 7.3 硬性规则，含 7.3.4 场景 C 附加规则）
  ↓
人工审校（按 7.5 审校要点，含翻译审校 + 双语审校）
  ↓
导出（zh.srt / en.srt / zh-en.srt / ASS）
```

### 双语排版方案

#### 方案 A：同一 cue 内双语

适合短视频、课程、内部培训。

```text
我们今天讨论交付时间。
Today we'll discuss the delivery timeline.
```

优点：时间轴简单，观众可同时对照
缺点：屏幕占用大，阅读速度容易超标

#### 方案 B：中英分轨

适合正式交付。

```text
zh-CN.srt
en-US.srt
```

优点：符合多数平台习惯，可单独审核每种语言，阅读体验更好

> **正式交付用中英分轨；预览、社媒、课程可额外生成双语 ASS。**

### 双语对齐优先级的参考

当原声是中文时，优先级建议：

```
中文语音时间轴 > 语义完整 > 英文可读性 > 中文显示舒适度 > 中英逐条对应
```

### cue 调整策略

1. **优先压缩目标语言**：原英文太长时，压缩为更简洁的表达
2. **合并源语言 cue**：源语言切得太碎时合并为更大的语义块
3. **拆分目标语言 cue**：目标语言太长但语义允许时，拆成两条
4. **微调时间轴**：允许小幅调整
5. **最后才改源语言表达**：仅当需要双语同步和显示舒适时，对源语言做轻度字幕化压缩（法律、采访、纪录片等忠实度要求高的内容除外）

# 第四部分：质量控制与导出

---

## 七、质量控制

### 7.1 转录质量

| 指标 | 目标 |
|---|---|
| 普通访谈 | 准确率 95% 以上 |
| 专业术语内容 | 需术语表 + 人工审校 |
| 人名地名品牌 | 必须人工确认 |
| 数字金额日期 | 必须人工确认 |
| 多人重叠说话 | 必须人工抽查或全查 |

### 7.2 说话人质量

| 项目 | 要求 |
|---|---|
| 说话人数量 | 自动识别后人工确认 |
| 说话人标签 | 从 `SPEAKER_00` 改为真实姓名 |
| 说话人切换 | 重点检查转场处 |
| 重叠说话 | 标记 `overlap` |
| 旁白 / 画外音 | 单独标记 |

---

### 7.3 硬性规则（必须通过，可自动检查）

#### 7.3.1 通用规则（所有场景）

| 规则 | 阈值 | 说明 |
|---|---:|---|
| 最大行数 | ≤ 2 行 | |
| 中文每行长度 | ≤ 16 个汉字 | |
| 英文每行长度 | ≤ 42 个字符 | |
| 成人阅读速度 | ≤ 9 字/秒（中文） | 英文按平台规范 |
| 儿童阅读速度 | ≤ 7 字/秒（中文） | 更保守 |
| 单条最短时长 | ≥ 0.8 秒 | |
| 单条最长时长 | ≤ 7 秒 | |
| 时间码重叠 | 不允许 | 相邻 cue 时间轴不能交叉 |
| 空字幕 | 不允许 | |
| 行末缺标点 | 不允许 | 中文句尾应有标点 |

#### 7.3.2 场景 A 附加规则（同语言字幕）

| 规则 | 说明 |
|---|---|
| 跨镜头切分 | 如台词未跨镜头，字幕应在镜头切换前结束 |
| 字幕闪烁间隔 | 相邻 cue 间隔不得低于最小安全值，或使用 chaining |

#### 7.3.3 场景 B 附加规则（目标语言单语字幕）

| 规则 | 说明 |
|---|---|
| 时间轴不越界 | 目标语言字幕时长不得超出源语言词级时间戳范围 |
| 关键术语一致性 | 人名、地名、数字、品牌名与源语言一致（抽检） |
| 目标语言阅读速度 | 严格按目标语言 CPS 上限执行 |
| 无遗漏翻译 | 语义 unit 覆盖完整，不存在漏译 |

#### 7.3.4 场景 C 附加规则（双语字幕）

| 规则 | 说明 |
|---|---|
| 双语 cue 映射完整性 | 无孤立的单语 cue（只有中文没有英文，或反之） |
| 合并阅读速度 | 双语同时显示时，每种语言独立不超速 |
| 视觉密度 | 单条字幕总字符数不宜过度密集 |
| 中英行数平衡 | 不应出现中文 1 行 + 英文 3 行的不平衡情况 |

---

### 7.4 柔性策略（推荐遵循，可据上下文灵活处理）

硬性规则之外，高质量字幕还需要考虑以下方面。这些策略不强制，但违反时应审慎确认是否有充分理由。

#### 7.4.1 语义完整性

- 每条 cue 尽量是一个完整的语义单元
- 跨多条 cue 的长句，每条应独立可读
- 避免将从句与其主句切分到不同 cue 中
- 数字、专名、术语应完整放在同一 cue 内

#### 7.4.2 换行自然度

- 双行字幕应在自然短语边界断行
- 避免第二行只有一个词（orphan word）
- 中文不拆：形容词 + 名词、动词 + 宾语
- 英文不拆：article + noun、preposition + object、auxiliary + main verb
- 复合词（如"人工智能"、"video production"）不应跨行

#### 7.4.3 时间轴舒适度

- `reading_padding` 不机械固定，慢速段落可增加，快速对话可减少
- 出点不应在语音结束后立即消失，留出约 0.2–0.5 秒眼动时间
- 如字幕结束在镜头切点但话音未停，可适当延后出点
- 每条 cue 的首字应尽量对齐对应词语的音节开始

#### 7.4.4 说话人一致性

- 一条 cue 应尽量只包含一个说话人
- 如需包含两人对话，每行一个说话人，前置说话人标签
- 跨说话人的长 cue 应优先拆分
- 说话人切换处应尽可能同步切分 cue

#### 7.4.5 镜头切点合理性

- 台词跨越镜头切点时，尽量在切点处调整 cue 边界
- 如台词确实连续跨镜头（如人物边走边说的长镜头），可不硬切
- 优先级：台词连续 > 不跨镜头

#### 7.4.6 翻译质量（场景 B/C）

- 译文应符合目标语言习惯，不拘泥字面
- 压缩优先于延长时间：译文太长时先压缩而非拉长时间轴
- 核心术语全文保持一致（同一概念在整片中译法统一）
- 语气、情感色彩应传递（不因压缩丢失原意）
- 填充词（well, you know, 就是说）、重复可省略

#### 7.4.7 双语协调（场景 C）

- 中文和英文应像一对孪生字幕，而非两条独立字幕
- 同一语义块的中英文行数尽量接近
- 如果一种语言远长于另一种，考虑整体拆分语义块
- 双语间不要逐字对齐，以语义块为单位对齐即可
